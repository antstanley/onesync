**Tech Stack:** `clap` v4 (already pinned), `tokio` for socket I/O, `serde_json`. Output colour via `owo-colors` (or built-in tracing) controlled by `NO_COLOR` env and `--no-color` flag.

## Task 2: `rpc/client.rs` — daemon connection

Connect to `${TMPDIR}onesync/onesync.sock` (resolved the same way the daemon resolves it). If connection refused → exit code 3 (Daemon not running) with a hint to start the daemon. If destructive operation requested but daemon not running, do NOT auto-start.

```rust
pub struct RpcClient {
    stream: tokio::net::UnixStream,
    next_id: AtomicU64,
}

impl RpcClient {
    pub async fn connect(socket: &Path) -> Result<Self, CliError>;
    pub async fn call<R: DeserializeOwned>(&mut self, method: &str, params: serde_json::Value) -> Result<R, CliError>;
    pub async fn subscribe<E: DeserializeOwned>(&mut self, method: &str, params: serde_json::Value) -> Result<SubscriptionStream<E>, CliError>;
}
```

Each request includes a fresh `id` (`req_<ulid>`) and the major version. On `version.major_mismatch`, exit code 10 with a clear upgrade hint.

**Tests:** unit tests against an in-process daemon (spawn one in `#[tokio::test]`).

**Commit:** `feat(cli/rpc): JSON-RPC client over Unix socket`

## Task 5: `exit_codes.rs` — stable exit codes

Implement the full table from spec 07 §CLI exit codes. The mapping function:

```rust
pub fn exit_code_for(err: &CliError) -> i32 {
    match err {
        CliError::Generic(_) => 1,
        CliError::InvalidArgs(_) => 2,
        CliError::DaemonNotRunning => 3,
        CliError::AuthRequired => 4,
        CliError::PairErrored => 5,
        CliError::ConflictUnresolved => 6,
        CliError::Permission(_) => 7,
        CliError::Network(_) => 8,
        CliError::LimitExceeded(_) => 9,
        CliError::VersionMajorMismatch => 10,
    }
}
```

`From<RpcError>` for `CliError` switches on `data.kind` (the machine identifier) to produce the right variant.

**Commit:** `feat(cli): exit code table per spec 07`

## Task 13: Auto-start logic

For non-destructive operations (`status`, `pair list`, `account list`, etc.): if connection refused, try `launchctl kickstart gui/$(id -u)/dev.onesync.daemon`, poll `health.ping` for up to 5 seconds, then retry. If still refused, exit code 3.

For destructive operations (anything with `--yes` semantics, plus `service` subcommands): never auto-start. Exit 3 with a clear message.

**Tests:** spawn an in-process daemon, then drive every command at least once.

**Commit:** `feat(cli): auto-start daemon for non-destructive ops`

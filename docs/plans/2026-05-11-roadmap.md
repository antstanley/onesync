# onesync — Implementation Roadmap

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement each milestone's plan task-by-task. This roadmap is the dependency map; per-milestone plans hold the executable steps.

**Goal:** Deliver the system described in [`docs/spec/`](../spec/) — a macOS daemon and CLI that two-way-syncs a designated local folder with a designated OneDrive folder, written in Safe Rust.

**Architecture:** Hexagonal — pure-logic core surrounded by adapter crates that implement port traits, two binaries (`onesyncd`, `onesync`) wire core to adapters. Nine crates total: see [`docs/spec/02-architecture.md`](../spec/02-architecture.md).

**Tech Stack:** Rust 1.95.0, Tokio, `rusqlite` (bundled), `notify` (FSEvents), `reqwest` (rustls), `blake3`, `security-framework`, `clap`, `serde`, `thiserror`, `proptest`. Test runner: `cargo-nextest`. Coverage: `cargo-llvm-cov`. VCS: jujutsu (colocated git).

  `RemoteDrive::item_by_path`, mint a `Pair` row.
- Daemon engine scheduler: long-running task that walks active pairs at
  `DELTA_POLL_INTERVAL_MS`, runs `engine::run_cycle` for each non-paused pair, drains on the
  `ShutdownToken`.
- `pair.force_sync`: enqueue a one-shot cycle through the scheduler; return
  `SyncRunHandle { run_id, started_at }`.
- `pair.status`: aggregate the Pair row, `runs_recent(pair, 5)`,
  `conflicts_unresolved(pair).len()`, and in-flight ops.
- `service.shutdown`: thread `ShutdownToken` into `DispatchCtx`; trigger on call with the
  `drain` flag.
- `state.backup` / `state.export` / `state.repair.permissions` / `state.compact.now`: wire
  each via existing helpers (`rusqlite::backup::Backup`, JSONL dump, chmod 0700/0600,
  `retention::run`).

Spec-decision implementations:

- *Symlink audit event* (decision in [`01-domain-model.md`](../spec/01-domain-model.md)): the
  scanner surfaces skipped symlinks via a callback the scheduler turns into per-pair
  `local.symlink.skipped` audit events. Today the skip happens in `onesync-fs-local::scan`
  without any audit emission.
- *Case-collision rename handling* (decision in
  [`01-domain-model.md`](../spec/01-domain-model.md)): scanner detects case-insensitive
  collisions on case-sensitive APFS volumes; the local-side challenger is renamed with a
  `(case-collision-<short-hash>).ext` suffix and flows through the normal pipeline; a
  `Conflict` row is recorded for review.
- *Webhook receiver via Cloudflare Tunnel* (decision in
  [`03-sync-engine.md`](../spec/03-sync-engine.md) and
  [`04-onedrive-adapter.md`](../spec/04-onedrive-adapter.md)): the daemon hosts an HTTP
  listener on a config'd local port; the Graph adapter learns to register `/subscriptions`
  for a pair when opted in; webhook deliveries push a trigger into the scheduler. Adds new
  fields to `InstanceConfig` (`webhook_listener_port`) and `Pair` (`webhook_enabled`) with
  matching migrations + schema entries.
- *Azure AD client registration* (decision in
  [`00-overview.md`](../spec/00-overview.md) and
  [`04-onedrive-adapter.md`](../spec/04-onedrive-adapter.md)): `InstanceConfig` grows an
  `azure_ad_client_id` field; `account.login.begin` reads it and refuses to start when it is
  unset. The install docs gain a section walking the user through the Azure AD registration
  form.

**Schema growth:**

`canonical-types.schema.json` grows two fields on `InstanceConfig` (`azure_ad_client_id`,
`webhook_listener_port`) and one on `Pair` (`webhook_enabled`). Each field is added with a
matching SQLite migration and a `serde(default)` so older state DBs upgrade cleanly.

**Out of scope (deferred to later milestones):**

- SharePoint document libraries (M11 — own milestone per the decision in
  [`04-onedrive-adapter.md`](../spec/04-onedrive-adapter.md)).
- Distribution + notarisation (release-engineering milestone per the decision in
  [`00-overview.md`](../spec/00-overview.md)).
- Subscription streaming push, upgrade flow, macOS host integration tests (M10).

## M11 — SharePoint document libraries

**Status:** Complete (`origin/main` @ TBD, 2026-05-13). Per the decision in [`04-onedrive-adapter.md`](../spec/04-onedrive-adapter.md#assumptions-and-open-questions). Pairs against SharePoint libraries now flow through the same `pair.add` path as personal-drive pairs; only the underlying `Account.drive_id` differs.

**Scope (landed):**

- `onesync_graph::items::site_by_path(http, token, host, site_path)` resolves a `SharePoint` site via `GET /sites/{host}:/sites/{path}` and returns the canonical Graph site id (`<host>,<site-guid>,<web-guid>`).
- `onesync_graph::items::site_library_by_name(http, token, site_id, library_name)` lists drives under the site and returns the first one whose display name matches.
- `account.add_sharepoint { base_account_id, host, site_path, library_name }` JSON-RPC method: refreshes the base account's access token, resolves the site + library, mints a new `Account` row reusing the base account's keychain entry (no extra login required), and emits an `account.added_sharepoint` audit event.
- `onesync account add-sharepoint` CLI subcommand wraps the RPC.

**Out of scope (intentional):**

- No new entity types. A SharePoint drive maps to `DriveId` + `DriveItemId` exactly like `/me/drive`, so the `Pair` schema is unchanged.
- The base account must already hold SharePoint-capable scopes (`Files.ReadWrite.All` typically). Tightening the install docs to walk the user through enlarging the scope set is left for M12.

**Exit criteria (met):**

- All workspace gates green.
- The CLI command exists and routes correctly to the daemon RPC.

## Spec-level decisions (resolved 2026-05-12)

The four open questions previously tracked here have been resolved. The full decisions live in
the relevant spec page's *Decisions* block; this list maps each to the milestone it affects.

1. **Azure AD client ID.** *User registers their own multi-tenant app*; onesync ships no
   project-owned client ID. Install docs walk through the registration form. See
   [`00-overview.md`](../spec/00-overview.md#assumptions-and-open-questions) and
   [`04-onedrive-adapter.md`](../spec/04-onedrive-adapter.md#assumptions-and-open-questions).
   Affects M3 (Graph adapter accepts the client ID from config rather than baking one in) and
   the install docs.
2. **Distribution channel.** *Homebrew formula and `curl | bash` from a notarised GitHub
   Release.* Both binaries are notarised + stapled so Gatekeeper does not prompt. See
   [`00-overview.md`](../spec/00-overview.md#assumptions-and-open-questions). Affects M7 and
   forthcoming release-engineering work.
3. **Webhook receiver.** *Cloudflare Tunnel terminates the `/subscriptions` callback URL;
   webhooks are opt-in, off by default; polling stays the always-on fallback.* See
   [`03-sync-engine.md`](../spec/03-sync-engine.md#assumptions-and-open-questions) and
   [`04-onedrive-adapter.md`](../spec/04-onedrive-adapter.md#assumptions-and-open-questions).
   Affects post-M8 work; the webhook receiver lands when the install docs ship a
   `cloudflared` config template.
4. **Case-sensitive APFS / name collisions.** *OneDrive's stored name wins; the local-side
   collider is renamed with a `(case-collision-<short-hash>)` suffix and synced as a new file.*
   See [`01-domain-model.md`](../spec/01-domain-model.md#assumptions-and-open-questions). This
   subsumes the previously separate case-sensitive APFS open question that lived in
   [`05-local-adapter.md`](../spec/05-local-adapter.md). Affects M2 (local adapter applies the
   rename) — schema changes not required.
5. **Large-file ceiling.** *`MAX_FILE_SIZE_BYTES = 10 GiB`* (dropped from the placeholder
   50 GiB). Lifting this is gated on hashing + upload-resume soak-test data. See
   [`00-overview.md`](../spec/00-overview.md#assumptions-and-open-questions). Constant landed
   in `crates/onesync-core/src/limits.rs`.
6. **SharePoint document libraries.** *Implemented in M11 (2026-05-13).* The engine does not
   need new entity types; the Graph adapter learned `site_by_path` + `site_library_by_name`,
   and the daemon added an `account.add_sharepoint` method + matching CLI subcommand. See
   [`04-onedrive-adapter.md`](../spec/04-onedrive-adapter.md#assumptions-and-open-questions).

## Findings

### F1. Adapter error mapping collapses every OS error to opaque `Io(String)` — BUG

**Location:** `crates/onesync-fs-local/src/adapter.rs:19-32` and `crates/onesync-fs-local/src/error.rs:26-30`
**Severity:** BUG
**Summary:** The port-level `LocalFsError` declares rich, engine-classified variants (`NotFound`, `PermissionDenied`, `NotMounted`, `DiskFull`, `QuotaExceeded`, `AlreadyRunning`), but the adapter's internal `LocalFsAdapterError` only carries `InvalidPath`, `Raced`, `CrossVolumeRename`, `Io(String)` — and `From<io::Error>` blindly stringifies into `Io`. Every real syscall failure (`ENOENT`, `EACCES`/`EPERM`, `EROFS`, `ENOSPC`, `EDQUOT`) reaches the engine as `LocalFsError::Io`.

**Evidence:**

```rust
// error.rs:26
impl From<std::io::Error> for LocalFsAdapterError {
    fn from(e: std::io::Error) -> Self { Self::Io(e.to_string()) }
}

// adapter.rs:19
fn map_err(e: LocalFsAdapterError) -> LocalFsError {
    match e {
        LocalFsAdapterError::InvalidPath { reason } => LocalFsError::InvalidPath { reason },
        LocalFsAdapterError::Raced => LocalFsError::Raced,
        LocalFsAdapterError::CrossVolumeRename { method } => {
            LocalFsError::CrossVolumeRename { method }
        }
        LocalFsAdapterError::Io(s) => LocalFsError::Io(s),
    }
}

// adapter.rs:30
fn map_io(e: &std::io::Error) -> LocalFsError { LocalFsError::Io(e.to_string()) }
```

Spec §"Error mapping" requires `ENOENT → NotFound`, `EACCES/EPERM → PermissionDenied`, `EROFS → NotMounted`, `ENOSPC → DiskFull`, `EDQUOT → QuotaExceeded`, `EEXIST → Exists`, `EXDEV → CrossVolumeRename`. None of these map through; the engine's pause-pair vs retry-with-backoff classification (e.g., disk-full pause until margin, permission-denied non-retryable) loses the input it depends on.

**Reasoning (certificate):**

```
PREMISES:
P1: Spec mandates classified port errors so the engine can branch behaviour.
P2: Adapter funnels all std::io::Error through From<io::Error> -> Io(String).
P3: map_err only knows InvalidPath, Raced, CrossVolumeRename, Io.

EXECUTION TRACE:
- unlink on a missing file -> std::io::Error{kind=NotFound}
  -> From<io::Error> -> LocalFsAdapterError::Io("...No such file...")
  -> map_err -> LocalFsError::Io("...")
  -> engine sees Io(...), not NotFound -> spec-mandated "treat as LocalSide=None" path is never reached.
- write to a full volume -> io::Error{kind=StorageFull / raw_os_error=ENOSPC}
  -> Io(String) -> engine retries instead of pausing the pair.
- read of a sandboxed path -> io::Error{kind=PermissionDenied}
  -> Io(String) -> engine retries (potentially forever) instead of surfacing
     non-retryable PermissionDenied per spec.

REGRESSION CHECK:
- Adding ErrorKind/raw_os_error classification in From<io::Error> would not
  break existing call sites; map_err already exhaustively matches the
  internal enum, so it must be widened in lock-step.

EDGE CASES:
- io::ErrorKind::StorageFull is stable since 1.83; raw_os_error() == Some(libc::EDQUOT)
  is the only portable QuotaExceeded probe.
- EROFS is read-only-filesystem, not unmount; mapping it to NotMounted as the
  spec table directs is debatable — but the spec is explicit, so adapter
  should follow it.

VERDICT: BUGGY (spec violation with observable engine effect).
CONFIDENCE: high.
```

**Suggested direction:** Inspect `io::Error::kind()` and `raw_os_error()` at the `From<io::Error>` boundary (or in a helper used by `write.rs`, `ops.rs`, `hash.rs`), expand `LocalFsAdapterError` (or skip the intermediate enum for non-classified failures and map straight into `LocalFsError`), and finalize the table in `map_err`. `AlreadyRunning` is daemon-startup territory and may live elsewhere — but the other six must land here.

### F2. Watcher emits non-NFC paths; scanner emits whatever the FS returned — BUG (spec violation)

**Location:**
- `crates/onesync-fs-local/src/watcher.rs:97-124` (`translate`)
- `crates/onesync-fs-local/src/scan.rs:72-127` (`scan_with_skips`)
- `crates/onesync-fs-local/src/adapter.rs:114-139` (`watch` adapter)
- `crates/onesync-protocol/src/path.rs:108-118` (`AbsPath::from_str` — NFC NOT applied)

**Severity:** BUG
**Summary:** Spec §"Path discipline" mandates NFC normalisation on the way in for everything the engine sees. `RelPath::from_str` normalises (`.nfc().collect()`), but `AbsPath::from_str` does **not** — and the watcher and scanner both emit raw `PathBuf` from the OS without converting. Files created on disk in NFD (the historical macOS HFS+ form, still common from older toolchains, USB drives, and AFP/SMB clients) reach the engine as NFD, do not match the NFC-normalised state-store keys, and the engine sees them as ghost files.

**Evidence:**

```rust
// onesync-protocol/src/path.rs:108
impl FromStr for AbsPath {
    fn from_str(s: &str) -> Result<Self, Self::Err> {
        if !s.starts_with('/') {
            return Err(PathParseError::NotAbsolute);
        }
        validate_common(s)?;
        Ok(Self(s.to_owned()))   // <-- no nfc() call (compare with RelPath line 102)
    }
}

// fs-local/src/watcher.rs:97
fn translate(event: &notify::Event) -> Option<LocalEvent> {
    let path = event.paths.first().cloned()?;
    match event.kind {
        EventKind::Create(_) => Some(LocalEvent::Created(path)),  // raw PathBuf
        ...

// fs-local/src/scan.rs:107
queue.push_back(path.clone());
out.push(LocalFileMeta {
    path,                                   // raw PathBuf, no NFC
    kind: FileKind::Directory, ...
});
```

`AbsPath` is the type the engine uses to index local-side state. If `AbsPath` doesn't normalise, two paths that the spec promises to alias map to two distinct keys.

**Reasoning (certificate):**

```
PREMISES:
P1 (spec): "two paths that differ only by Unicode normalisation map to the same RelPath... normalise on the way in".
P2 (port): scan returns Vec<(PathBuf, FileSide)>, NOT Vec<(AbsPath, ...)>.
P3 (port): LocalEventDto carries PathBuf, NOT AbsPath.
P4: AbsPath::from_str does not normalise.

EXECUTION TRACE — NFD file already on disk:
- User has /Users/alice/OneDrive/café.txt stored on disk in NFD ("cafe\u{0301}.txt").
- Engine scans -> scan_with_skips returns PathBuf in NFD.
- Engine writes /Users/alice/OneDrive/café.txt in NFC via write_atomic ->
  spec promises rename(NFC, NFD) -> overwrites the NFD form on APFS (case-preserving,
  Unicode-equivalent for HFS+ history, but APFS treats them as DISTINCT bytes!).
- Result: two files differing only by normalisation form coexist on disk.
  Spec said this is forbidden.

EXECUTION TRACE — watcher rename event:
- Renamed { from: <NFC>, to: <NFD> } may arrive depending on which tool wrote it.
- Engine compares from against state-store key (NFC) -> miss.
- Treated as Created(<NFD>); the rename heuristic is defeated.

REGRESSION CHECK:
- Adding nfc to AbsPath::from_str: callers that currently construct AbsPath from
  user-provided strings keep working; adapter's PathBuf -> AbsPath bridge is
  the spot that benefits.
- The scan/watcher boundary still hands out PathBuf, so a sibling change is
  needed: normalize the PathBuf's UTF-8 representation before handing to the
  engine. The port-type itself uses PathBuf — that's a port-design hazard
  (see F8).

EDGE CASES:
- Volume-root path containing NFD components is silently accepted and propagated.
- LocalEvent::Renamed.{from,to} are independently denormalised.

VERDICT: BUGGY (spec violation, latent until first NFD-bearing file appears).
CONFIDENCE: high.
```

**Suggested direction:** (a) NFC-normalise in `AbsPath::from_str` (symmetry with `RelPath`). (b) Convert `PathBuf` to a normalised string at the watcher/scanner boundary, then re-construct `PathBuf` from the NFC string before handing to the engine. (c) Add a regression test that seeds an NFD-named file on disk and asserts the adapter surfaces the NFC form.

### F3. Watcher `EventKind` matcher falls through silently on `Access` and `Any` — CONCERN

**Location:** `crates/onesync-fs-local/src/watcher.rs:97-124`
**Severity:** CONCERN
**Summary:** `translate` matches `Create(_)`, `Modify(Name(_))`, `Modify(_)`, `Remove(_)`, `Other`, and the wildcard `_ => None`. `notify::EventKind::Access(_)` and `notify::EventKind::Any` both fall through to `_ => None`. `Access` is intended (the watcher does not care about reads). But `Any` is the notify-side "platform produced an event we couldn't classify" sentinel — dropping it silently means a legitimate change can disappear with no overflow signal.

**Evidence:** lines 99-123. There is no explicit `EventKind::Any => Some(LocalEvent::Overflow)` (or equivalent), and `Other` is documented as containing potential `MustScanSubDirs` markers that the adapter explicitly notes "Treat as no-op for now" (line 119) — that's a deliberate hole that the spec says should be `Overflow`.

**Reasoning:** The wildcard match keeps the matcher exhaustive (compiles), but the price is that any future `EventKind` variant added by `notify` and any platform-quirk `Any` event is silently dropped. The spec is clear: `MustScanSubDirs` is `Overflow`. The current code maps it to `None`.

**Suggested direction:** Explicit arms for `Access(_) => None` (and document why), `Any => Some(Overflow)`. Once `notify` exposes the `MustScanSubDirs` flag in `Other`'s payload (or via an explicit variant), wire that through too.

### F4. `try_send`-on-full silently drops events and then fights itself for the `Overflow` slot — CONCERN

**Location:** `crates/onesync-fs-local/src/watcher.rs:73-95`
**Severity:** CONCERN
**Summary:** When the bounded FSEvents channel (`FSEVENT_BUFFER_DEPTH` = 4096) is full, `tx.try_send(evt).is_err()` returns; the code then immediately attempts `tx.try_send(LocalEvent::Overflow)`. But the channel is full — so the second `try_send` also fails. The `let _ =` discards both errors. The consumer never observes an `Overflow` event. Subsequent successful `try_send`s after a partial drain will pass through normal events ahead of any overflow signal, so ordering is wrong too.

**Evidence:**

```rust
if let Some(evt) = local
    && tx.try_send(evt).is_err()
{
    // Channel full or closed — best-effort signal overflow.
    let _ = tx.try_send(LocalEvent::Overflow);
}
```

The comment acknowledges "best-effort". The spec promises a single `Overflow` flag after drop; this implementation cannot deliver that promise when the consumer is genuinely behind by ≥4096 events. There is no sticky `overflow_pending` flag carried across callback invocations.

**Reasoning (certificate):**

```
PREMISES:
P1 (spec): "When the channel is full, the adapter sets the Overflow flag and
            drops the event (the engine will re-scan)."
P2: try_send on a full bounded channel returns Err(Full) immediately.
P3: There is no out-of-band overflow flag; the only signal is a LocalEvent::Overflow
    enqueued in the same channel that is currently full.

EXECUTION TRACE — sustained overload:
- Channel saturates at 4096.
- Event #4097 arrives -> try_send(evt) Err; try_send(Overflow) Err. Both dropped.
- Consumer eventually drains 100 events; capacity now 100.
- Event #4197 arrives -> try_send(evt) Ok. Engine processes a stale event that
  came AFTER a window of dropped events; engine still thinks watcher is healthy.
- The engine NEVER sees an Overflow and NEVER triggers a re-scan.

REGRESSION CHECK / SUFFICIENCY:
- Would fixing ONLY the try_send-on-full path fix the symptom?
  No: the design has no out-of-band overflow channel. A correct fix uses
  an AtomicBool "overflow_pending"; when set, the next successful try_send
  (or a periodic flusher) inserts a LocalEvent::Overflow first.

EDGE CASES:
- During Unmounted teardown, the order of "drain remaining events" vs
  "emit Unmounted" is not enforced; an Unmounted may be dropped the same
  way during burst.

VERDICT: CONCERN (correctness gap matching real long-soak scenarios; not yet
demonstrably breaking a test).
CONFIDENCE: medium-high.
```

**Suggested direction:** Track overflow in an `AtomicBool` (or a shared cell) on the closure's captured side. When set, the next opportunity to enqueue (either a successful normal `try_send` or a wake-up from a periodic timer) inserts a single `Overflow` first, clearing the flag. Alternatively, reserve one slot of headroom — when channel occupancy reaches `cap - 1`, queue Overflow proactively.

### F5. Hash race window: open happens between the two `mtime` probes — BUG (false negative on mid-hash mutation)

**Location:** `crates/onesync-fs-local/src/hash.rs:23-44`
**Severity:** BUG
**Summary:** The mtime-equality check has a gap: `mtime_before` is read, then `File::open` runs, then bytes are streamed, then `mtime_after` is read. If a writer renames a new file over the path between `mtime_before` and `File::open`, we hash the new file but compare mtimes of the new file to the new file — `mtime_before` == `mtime_after`, no `Raced`. We return the hash of the wrong content with no error.

Worse: there's no `inode`/`size` check. Even within the existing window, writer activity that completes within mtime granularity (1 second on HFS+, 1 ns on APFS but coalesced) is undetectable.

**Evidence:**

```rust
pub fn hash(path: &Path) -> Result<ContentHash, LocalFsAdapterError> {
    let mtime_before = mtime(path)?;          // STAT #1 on path

    let mut file = File::open(path)?;         // OPEN #2 on path — may bind to a NEW inode
    ...
    let mtime_after = mtime(path)?;           // STAT #3 on path — may be the new file's mtime
    if mtime_before != mtime_after { Raced }  // checks the wrong invariant
    ...
}
```

The correct pattern: `open` first; on the open `fd`, `fstat` to get `mtime_before` and `inode_before` and `size_before`; stream through the `fd` (no re-resolution of the path); `fstat` again at the end; compare all three on the same fd.

**Reasoning (certificate):**

```
PREMISES:
P1: rename(2) and unlink+create are atomic but produce a NEW inode at `path`.
P2: hash() resolves `path` three times: stat, open, stat.
P3: Spec: "Raced if mtime changed during hashing".

EXECUTION TRACE — writer racing scan:
T0: scan finds file F with mtime m1.
T1: hash() opens => mtime_before = m1 (the ORIGINAL inode).
T2: writer: write_atomic completes; rename installs a new inode N2 with mtime m2.
T3: hash() does File::open(path) -> opens N2 (the new inode).
T4: hash streams N2's content (length L2, hash H2).
T5: hash stats path -> mtime_after = m2.
RESULT: H2 returned, hash succeeds, but mtime_before (m1, stale read of N1)
        is COMPARED to mtime_after (m2, stat of N2). m1 != m2 -> Raced.
        OK in THIS trace, but only because we got lucky: if m1 happened
        to equal m2 (clock resolution coincidence on coarse FS, or NTP
        backslip), no Raced.

Trace #2 — writer winning the open race fully:
T0: stat -> mtime_before = m1
T1: writer atomic-replaces in this window -> new inode N2 with mtime m2
T2: open(path) -> N2
T3: stream N2
T4: stat -> mtime_after = m2
T5: m1 != m2 -> Raced returned. OK.

Trace #3 — writer ALSO wins both stats (worst case):
T0: writer replaces => N2 with m2
T1: stat -> mtime_before = m2 (we already missed the original)
T2: open -> N2
T3: stream -> hash of N2
T4: stat -> mtime_after = m2
RESULT: returns hash(N2) silently. CALLER WAS HASHING N1; got H2 anyway.
        Caller thinks they have a valid hash for the file they sampled
        at scan time. WRONG.

REGRESSION CHECK / SUFFICIENCY:
- Fixing the comparison alone (using fstat on the open fd) closes the
  in-flight race fully. The fd binds to a specific inode; subsequent
  rename(2) of the path does not affect the fd.

EDGE CASES:
- Empty file: open succeeds, read returns n=0 immediately, mtime check
  may always pass (no opportunity to race). Less risky.
- File deleted mid-hash: open returns ENOENT correctly.

VERDICT: BUG (race produces false-negative for Raced detection).
CONFIDENCE: high.
```

**Suggested direction:** `let file = File::open(path)?;`, then `let m1 = file.metadata()?.modified()?;`, stream via `file`, then `let m2 = file.metadata()?.modified()?;` and compare. Also capture `dev`+`ino`+`size` for a stronger invariant. Bonus: this also lets you detect a truncate that happened to preserve mtime.

### F6. `read` slurps the entire file into a `Vec<u8>` — CONCERN (memory hazard at 10 GiB cap)

**Location:** `crates/onesync-fs-local/src/adapter.rs:67-74`
**Severity:** CONCERN
**Summary:** `std::fs::read(&pb)` loads the entire file at once. `MAX_FILE_SIZE_BYTES = 10 * GIB`. A single upload of a 10-GiB file blows the daemon's RAM. The port docstring acknowledges "eager; streaming Read comes later" — so this is a known limitation, but the spec §"Responsibilities #4" explicitly says **"Read a file as a streaming source the engine can pipe into uploads"**, and the upload-session machinery (`SESSION_CHUNK_BYTES = 10 MiB` per spec/limits.rs) expects streaming.

**Evidence:**

```rust
async fn read(&self, path: &AbsPath) -> Result<LocalReadStream, LocalFsError> {
    let pb = PathBuf::from(path.as_str());
    let bytes = tokio::task::spawn_blocking(move || std::fs::read(&pb))
        ...
    Ok(LocalReadStream(bytes))
}
```

`LocalReadStream(pub Vec<u8>)` ratifies the choice at the port level. Until both port and impl move to a stream type, the OneDrive adapter cannot stream uploads from disk; it must materialise the whole file twice (once on local read, once on Graph SDK chunking).

**Suggested direction:** Stage 1: enforce `MAX_FILE_SIZE_BYTES` at the port and refuse files larger than a small cap (e.g., 256 MiB) until streaming is in. Stage 2: change `LocalReadStream` to wrap an `impl AsyncRead`. The port comment already foreshadows this.

### F7. `write_atomic` does not preserve original file permissions; tempfile defaults to 0600 — CONCERN

**Location:** `crates/onesync-fs-local/src/write.rs:32-57`
**Severity:** CONCERN
**Summary:** `tempfile::NamedTempFile::new_in(parent)` creates a fresh file with `O_EXCL` and mode `0600` (tempfile default). When `persist()` does a `rename(2)`, the resulting file has mode `0600` even if the file it replaced had mode `0644` (or `0664`, etc.). This is a silent permission downgrade for any path the user previously made group-readable or world-readable. The spec §"Atomic writes" does not call out preservation, but `local.metadata.dropped` is meant to be emitted for xattrs/ACLs/resource-forks — POSIX permission bits are a notable additional metadata loss with no audit signal.

**Evidence:** lines 32, 51-52 (`NamedTempFile::new_in` then `persist`). No `set_permissions` call against the target's prior mode.

**Reasoning:** This is correctness-by-spec ambiguity rather than a hard bug; the spec doesn't require permission preservation, but does require auditing dropped metadata. The current behaviour silently changes file modes during sync, which can break web-server-served directories, shared media, etc.

**Suggested direction:** Stat the target before write_atomic; on success, `fchmod` the temp file fd to match the target's prior mode (or default `0644` for new files), emit `local.metadata.dropped` if the prior mode was anything weirder (setuid/setgid/sticky). At minimum, document the behaviour in the spec.

### F8. Cross-volume `rename` fallback skips fsync and reports as `Err` despite success — BUG

**Location:** `crates/onesync-fs-local/src/ops.rs:15-28`
**Severity:** BUG
**Summary:** Two distinct bugs in the cross-volume rename fallback:

1. **No fsync**: spec §"Path discipline" says "each step fsync'd", but `std::fs::copy` does not fsync, and there is no parent-directory fsync before returning. A crash mid-fallback can leave a partially-copied destination and the source still present, or the destination present and the source still present despite both syscalls returning OK (write-back buffer never flushed).
2. **Returns `Err(CrossVolumeRename)` even on success**: the doc comment says "the caller treats this as an audit signal, not a fatal error". But the only consumer is `adapter::map_err` which forwards it to `LocalFsError::CrossVolumeRename`. `CrossVolumeRename` is *defined* in the port type (`crates/onesync-core/src/ports/local_fs.rs:29`) but **not referenced anywhere else in `onesync-core` or `onesync-daemon`** (verified by grep). The engine sees it as a generic failure variant. The "audit signal" path is not wired.

3. **No cleanup on partial failure**: if `copy` succeeds but `remove_file` fails (e.g., source EACCES), both files exist and the error is reported with no indication that the destination is now also present.

**Evidence:**

```rust
pub fn rename(from: &Path, to: &Path) -> Result<(), LocalFsAdapterError> {
    if from.exists()
        && to.parent().is_some_and(Path::exists)
        && !same_volume(from, to.parent().unwrap_or(from))
    {
        // Cross-volume: copy + delete, then surface the degradation.
        std::fs::copy(from, to)?;          // no fsync of destination fd
        std::fs::remove_file(from)?;       // if this fails, dest is "ghosted"
        return Err(LocalFsAdapterError::CrossVolumeRename { method: "copy+delete" });
                                            // ↑ semantically Ok, returned as Err
    }
    std::fs::rename(from, to).map_err(LocalFsAdapterError::from)
}
```

Compare with `write_atomic` which does sync the file and the directory.

**Reasoning (certificate):**

```
PREMISES:
P1 (spec): Cross-vol rename degrades to copy+delete "with each step fsync'd".
P2 (port doc): CrossVolumeRename is an audit hint, not a fatal failure.
P3 (grep): CrossVolumeRename variant has zero engine-side consumers.

EXECUTION TRACE — typical cross-vol move:
- copy(from, to): bytes land in page cache; no fsync.
- remove_file(from): inode released.
- Return Err(CrossVolumeRename).
- adapter::map_err -> LocalFsError::CrossVolumeRename
- Engine: matches against its error-handling logic, finds no special case
  for CrossVolumeRename, falls through to a generic "rename failed" branch
  -> retry? planner-state still thinks source exists at `from`? Diverges
  from the actual filesystem state.

EXECUTION TRACE — crash after copy but before remove:
- copy(from, to): bytes in page cache.
- Crash.
- On restart: from still exists (delete didn't run); to may or may not
  exist depending on which volume's page cache flushed (it had no fsync).
- Engine scans: sees from at original path, sees to perhaps. Plans
  another rename. Source content survives, so the second attempt may
  also "succeed". File-id divergence likely if remote also moved.

REGRESSION CHECK:
- Returning Ok(()) when degradation happened is the right semantic; the
  audit signal can be emitted via the audit sink (not via Result::Err).
- Adding sync_all to the destination fd before close (or via
  std::fs::OpenOptions::new().write(true).read(true)... and a manual
  read+write loop with sync_all) closes the durability gap.

VERDICT: BUG (durability gap + result-vs-success contract mismatch).
CONFIDENCE: high.
```

**Suggested direction:** Replace `std::fs::copy` with an explicit `OpenOptions::create_new(true)` + read/write loop that fsyncs the destination fd before close; fsync the destination's parent directory; on success of both, **return `Ok(())`** and emit the audit signal through the audit sink instead of via the result. Decide what to do if `remove_file(from)` fails: roll back by removing `to`, or surface a distinct error variant.

### F9. `same_volume` TOCTOU on cross-volume detection — CONCERN

**Location:** `crates/onesync-fs-local/src/path.rs:66-72` and `ops.rs:18`
**Severity:** CONCERN
**Summary:** The `same_volume` check stats both paths before the rename. Between the check and the rename(2), a volume could be remounted (network share reconnect on a different mount point, or a USB drive remount) and the rename could now span volumes. `rename(2)` returns `EXDEV` in that case, which the current code maps to generic `Io` (see F1) — losing the chance to retry via the copy+delete fallback.

Conversely, `same_volume` returns `false` when either path doesn't exist (e.g., destination parent missing). The fallback `to.parent().unwrap_or(from)` (line 18 of ops.rs) means we check `from`-vs-`from` if `to.parent()` is `None` — which is wrong on a true cross-volume situation. The chain of `is_some_and(Path::exists) && !same_volume(...)` mostly mitigates this but it's fragile.

**Suggested direction:** Try `std::fs::rename` first; on `EXDEV` (raw_os_error 18), fall back to copy+delete. This pattern eliminates the TOCTOU and is the conventional approach. It also lets you delete the `same_volume`/`metadata().dev()` probe entirely.

### F10. Scanner ignores spec-mandated `MAX_PATH_BYTES`, `SCAN_INFLIGHT_MAX`, and `Icon\r`-class skip list — CONCERN

**Location:** `crates/onesync-fs-local/src/scan.rs:24, 72-127`
**Severity:** CONCERN
**Summary:** Spec §"Scanning" promises:

- "Skips files whose absolute path length exceeds `MAX_PATH_BYTES` (1024) with a warning." — not implemented (no length check on `path` before `out.push`).
- "Producer fills up to `SCAN_INFLIGHT_MAX` entries and back-pressures." — `scan_with_skips` is fully synchronous; output `Vec<LocalFileMeta>` is unbounded. A 1 M-file pair root produces 1 M `LocalFileMeta` (~each ~100 bytes -> ~100 MiB of heap) with no back-pressure.
- `SKIP_NAMES` is `[".DS_Store", "Icon\r", ".localized"]` — present. But the spec also enumerates `._*` (handled). Other typical macOS noise like `.fseventsd`, `.Spotlight-V100`, `.Trashes`, `.TemporaryItems` is not — these usually exist at volume roots and not in user folders, but a top-level OneDrive folder *can* contain `.DocumentRevisions-V100` etc. if a user mounted the wrong path; the deny list silently includes them in scans.
- Spec calls out a `SKIPPED_LOCAL_NAMES` constant; `scan.rs:24` hardcodes `SKIP_NAMES` instead. Refactor target.

**Reasoning:** None of these is yet a demonstrable crash bug, but they widen the gap between spec and implementation. `MAX_PATH_BYTES` in particular matters because `AbsPath::from_str` enforces it strictly (`PathParseError::TooLong`), so a scanned file whose name overflows could parse-fail at the engine boundary — landing as a generic adapter error rather than an explicit "path too long" skip with audit.

**Suggested direction:** Add a length check after building `path` in scan.rs; emit a skip record. Move the hard-coded list into `onesync-core::limits` (or a fs-local const named `SKIPPED_LOCAL_NAMES`) and align with the spec. Convert `scan_with_skips` to async with a bounded `mpsc` of `SCAN_INFLIGHT_MAX`, or document the deviation.

### F11. Watcher-scanner race: events emitted before scan completes are not buffered by the adapter — CONCERN

**Location:** `crates/onesync-fs-local/src/adapter.rs:36-65, 114-139`; conceptual gap across the watch/scan pair.
**Severity:** CONCERN
**Summary:** The engine's initial reconcile pattern is "start watching, then full-scan". If a file is created between `watcher::watch()` returning and `scan()` enumerating that file's parent, the watcher emits a `Created` for the file and the scan also emits the file. So far so good (duplicate is fine). But consider a file that is **created then deleted** in this window:

- T0: `watch()` registered; FSEvents stream live.
- T1: `scan()` starts walking the root, queue contains `/root`.
- T2: user creates `/root/sub/new.txt` (watcher emits Created).
- T3: scan reaches `/root/sub`, but the user deletes `/root/sub/new.txt` (watcher emits Deleted).
- T4: scan's `read_dir(/root/sub)` no longer sees `new.txt`. Scan finishes with no entry for it.
- T5: Engine reads scan -> nothing about new.txt.
- T5: Engine reads events -> sees `Created(new.txt)` then `Deleted(new.txt)`.

If the engine processes scan as authoritative baseline and watcher as delta, the events sequence is harmless. **But the inverse race exists**: file enumerated by scan, then deleted before the engine touches it, with the `Deleted` event arriving before scan results are consumed:

- T0: file exists.
- T1: scan emits it.
- T2: watcher emits `Deleted` (some external tool).
- T3: scan completes; engine sees the file in scan results but Deleted in events.

This is the usual "watcher-then-scan vs scan-then-watcher" debate. The adapter does **not** document or implement an ordering guarantee; both streams are produced independently. The engine must reconcile by re-querying the FS, but the spec does not enforce this and the adapter does not surface a "scan baseline timestamp" to anchor event-vs-scan ordering.

**Reasoning:** This is a known design tension; spec §"Open questions" does not mention it. Worth flagging because it affects correctness if the engine relies on event-vs-scan ordering.

**Suggested direction:** Document the contract: which stream is authoritative at startup. A common pattern is "scan provides a baseline mtime watermark; events with mtime > watermark are authoritative". Alternatively, hold all events received during scan in a buffer and replay them after scan completes; the adapter could provide that as an opt-in.

### F15. `fakes.rs` diverges from real adapter semantics (no NFC, no case-insensitivity, ordering not modeled) — CONCERN

**Location:** `crates/onesync-fs-local/src/fakes.rs:39-230`
**Severity:** CONCERN
**Summary:** The in-memory fake uses `HashMap<PathBuf, FakeFile>` keyed by raw `PathBuf`. APFS is case-insensitive (default volume) and Unicode-equivalent under NFC vs NFD. The fake treats `/root/A.txt` and `/root/a.txt` as distinct keys; treats `/root/café.txt` (NFC) and `/root/cafe\u{0301}.txt` (NFD) as distinct keys. Engine tests using the fake therefore cannot exercise the real "case-insensitive collision" path or the real "NFC alias" path.

Per F2, this is also where the discrepancy is masked: real-adapter tests don't get NFC issues because they don't go through state-store keys, and fake tests don't because the fake never models the equivalence.

Additionally:
- `inject_event` sends to the most recent watcher; multiple watchers race for the slot.
- `inject_event` writes to a channel without modifying `files`, so a `Created` event can refer to a path that doesn't exist in the fake's filesystem — fine for some tests, surprising for others. Document explicitly.

**Suggested direction:** Use a normalisation key (NFC-lower-cased string) for the HashMap; store the original-cased + original-form path inside `FakeFile` for retrieval. Mirrors APFS's "preserve form, fold for lookup" semantics. Add a test that asserts `seed_file("/A")` then `read("/a")` returns the seeded contents (or document that the fake intentionally diverges, with rationale).

### F16. Adapter `watch` task leaks if the FSEvents `Watcher` outlives the channel consumer — NIT

**Location:** `crates/onesync-fs-local/src/adapter.rs:114-139`
**Severity:** NIT
**Summary:** `adapter::watch` spawns an anonymous task that pumps from the inner `Watcher` into a fresh mpsc; when the consumer drops the `LocalEventStream`, `tx.send(dto).await.is_err()` returns and the task exits. Good. But the `Watcher` (held inside `w` captured by the spawned task) is dropped only when the task exits, which happens only when the inner watcher's `recv()` returns `None` OR the outer `tx` errors. There is no explicit shutdown signal; the spawned task is detached. Dropping the inner Watcher also cancels FSEvents, so it's mostly fine — but the task remains pending until the next event arrives. On a quiet root, the task can sit indefinitely after the engine drops the stream.

**Suggested direction:** Use `tokio::select!` with a shutdown hook, or wrap the bridge task's lifetime in a `JoinHandle` returned alongside the stream (cancel on the engine's pair-stop path).

### F18. Watcher rename pairing: macOS `notify` typically emits split From/To events; current code degrades both to `Modified` — CONCERN

**Location:** `crates/onesync-fs-local/src/watcher.rs:101-113`
**Severity:** CONCERN
**Summary:** The implementation comment (lines 5-9) acknowledges that when `event.paths.len() < 2`, the renamed event "falls back to emitting a `Modified` event" and "engine-side reconciliation can re-pair such events by content hash". In practice on macOS, `notify` 6.x usually delivers `Name(From)` and `Name(To)` as **two separate events with one path each** — meaning the `event.paths.len() >= 2` branch is the rare case, and the common case is two `Modified` events (one for the source path, one for the destination path).

The downstream engine then sees `Modified("old.txt")` followed by `Modified("new.txt")`. Without other signals, the engine cannot distinguish this from "old.txt mutated and new.txt also mutated independently". The fallback to "re-pair by content hash" is doable but requires the engine to actually run hash comparison and history reasoning. The adapter could instead expose `LocalEvent::RenameFromPathHint(p)` / `RenameToPathHint(p)` to carry the `notify::event::RenameMode` information forward.

**Suggested direction:** Match on `ModifyKind::Name(RenameMode::From)` / `Name(RenameMode::To)` / `Name(RenameMode::Both)` distinctly. Emit `LocalEvent::RenameFromHint(path)` and `LocalEvent::RenameToHint(path)` (or include rename-mode as a side-channel on `Modified`). The engine then has the structural information it needs without a content-hash heuristic.

## Cross-cutting observations

- **Spec-vs-impl drift is substantial.** F2 (NFC), F8 (cross-vol fsync), F10 (length cap, back-pressure, skip list), F14 (volumes) are all spec sections with no implementation. F1 (error mapping) is a spec table that the impl ignores wholesale.
- **No `unsafe` in scope.** `lib.rs:5` `#![forbid(unsafe_code)]` is honoured; no FFI shims in this crate. `same_volume` uses `std::os::unix::fs::MetadataExt::dev` which is the Safe Rust path. No `panic!`/`todo!`/`unimplemented!` in non-test code (verified by inspection).
- **`expect`/`unwrap` outside tests:** one in `path::absolutise` (line 56), justified by a lint comment ("composed path is absolute and validated"). That argument holds if `RelPath` validation is correct, but the validation is **on `RelPath`**, and `RelPath` does NFC-normalise. The composed `AbsPath` may therefore differ from the input prefix's normalisation form (since `AbsPath::from_str` does not normalise — F2). The expect cannot panic on validity grounds, but the resulting `AbsPath` may not round-trip through `relativise`.
- **Read path bypasses `MAX_FILE_SIZE_BYTES`.** No size check in `read`, `hash`, or `write_atomic`. The `MAX_FILE_SIZE_BYTES = 10 GiB` cap is documented in `limits.rs` and consumed (per grep) only by `remote_drive.rs`. The local side should enforce it pre-read to avoid an OOM.
- **No `O_CLOEXEC`-flagged file opens.** Spec §"Atomic writes" promises "the temp file is created with the same `O_CLOEXEC` flag the daemon uses elsewhere". `tempfile::NamedTempFile` does set `O_CLOEXEC` by default on Unix (verify via the `tempfile` crate docs), so this is probably OK — but no code in this crate explicitly sets it, so a regression in `tempfile` defaults would silently flip the behaviour.
- **`spawn_blocking` is used for every entry point.** Good for cooperative scheduling. The flip side: `scan` is fully synchronous on the blocking thread, so a slow disk pegs one Tokio blocking-worker for the whole scan duration (no `read_dir` interleaving). For the M2 scale this is fine, but a per-directory yield (`tokio::task::yield_now().await` in an async scanner) is worth keeping on the radar.
- **Drop order matters but isn't enforced.** `Watcher { _watcher, rx }` (`watcher.rs:43`) — `_watcher` field comes first, so dropping the `Watcher` drops the inner notify `_watcher` first, then `rx`. That's the safe order (cancel the producer before closing the receiver). Currently relies on Rust's structural field-drop order; an explicit `Drop` impl would document the intent.

## What looks correct

- `lib.rs:5` `#![forbid(unsafe_code)]` — workspace rule honoured.
- BLAKE3 streaming uses `HASH_BLOCK_BYTES = 1 MiB` chunks — matches spec.
- Empty-file hash test (`hash_of_empty_file_matches_blake3_of_empty`) pins the documented digest — good fixture.
- `write_atomic` correctly: (a) creates temp in the target's directory (rename stays on-volume), (b) fsyncs the temp fd, (c) `persist`'s rename, (d) fsyncs the parent directory. Order matches the spec's recipe step-for-step.
- `RelPath` validation in `onesync-protocol` does NFC, rejects `..`, NUL, empty, leading slash, oversize — solid.
- `relativise` rejects path == root explicitly (returns `InvalidPath`) — prevents an empty `RelPath`.
- `scan` skip-list catches `.DS_Store`, `Icon\r`, `.localized`, and the `._*` prefix — partial spec coverage.
- Cross-volume detection via `dev()` comparison is the canonical Unix approach (modulo the TOCTOU in F9).
- `Watcher::recv` is `async` and returns `Option`, surfacing the "stream ended" condition cleanly.
- Tests for `path::same_volume`, `absolutise`, `relativise` cover the happy and unhappy paths; `scan` tests cover symlinks, sidecars, `.DS_Store`, recursion depth, mtime sanity.
- `adapter_round_trips_a_scan_write_read_delete_pipeline` is a good end-to-end smoke test.

## Findings

### F1. Spec violation: `ReAuthRequired` does not delete the stale keychain entry — BUG
**Location:** `crates/onesync-daemon/src/scheduler.rs:582-626` (`VaultBackedTokenSource::handle_re_auth_required`); compare against `docs/spec/04-onedrive-adapter.md:94-96`.
**Severity:** BUG
**Summary:** The spec mandates that on `invalid_grant` from the refresh endpoint, the adapter MUST delete the keychain entry and clear the in-memory cache. The scheduler's `VaultBackedTokenSource` does neither — it marks pairs as `Errored("re-auth required")` and emits an audit event, but the dead refresh token stays in the keychain and (if anything happens to be in the in-memory `cache: Mutex<Option<CachedToken>>`) the stale access token survives.
**Evidence:**
- Spec line 94: *"On HTTP 400 `invalid_grant` from the refresh endpoint, the adapter deletes the keychain entry, clears the in-memory token cache for that account, and surfaces `GraphError::ReAuthRequired`."*
- `handle_re_auth_required` (scheduler.rs:582) iterates pairs, calls `pair_upsert`, emits an audit event, and returns — no `self.vault.delete(...)` and no `self.cache.lock().await.take()`.
- The caller (`access_token`, scheduler.rs:657-660) re-propagates `ReAuthRequired` without clearing.
**Reasoning (fault-localization):**
- P1 (symptom): after Microsoft revokes a refresh token, the keychain still holds the revoked secret.
- P2 (expected): keychain entry deleted; cache cleared.
- D1 at scheduler.rs:582-626: only mutates state-store rows, never touches `self.vault` or `self.cache`. Sufficient to fix the symptom? Yes — adding `let _ = self.vault.delete(&self.account_id).await;` and clearing `self.cache` here would satisfy the spec.
- Consequence: next daemon restart (or next scheduler tick) will reload the same dead refresh token and re-trigger the same revoke → unbounded retry-storm; user-visible behaviour matches "auth broken" but observability never improves because we keep hitting the same wall.
**Suggested direction:** In `handle_re_auth_required`, before flipping pair status: delete the vault entry for `self.account_id`, replace the cache with `None`. Also consider whether `account.login.await` should re-bind the same `AccountId` (likely not — a fresh login mints a new id) so deletion is safe.

### F2. `LoginRegistry` has no TTL / no sweep — CONCERN
**Location:** `crates/onesync-daemon/src/login_registry.rs:28-62` and `crates/onesync-daemon/src/methods/account.rs:124-131`.
**Severity:** CONCERN
**Summary:** Sessions inserted by `account.login.begin` are removed only by an explicit `account.login.await(handle)`. If the user closes the CLI before calling `await`, the `LoginSession` (which owns a `oneshot::Receiver` plus `pkce_verifier` and `redirect_uri`) lives in the `HashMap` for the lifetime of the daemon. PKCE verifier and a oneshot resolved with the actual auth code can sit in memory indefinitely.
**Evidence:** `LoginRegistry` exposes only `insert` and `take`; the type carries no expiry timestamp; no background sweep task is wired up (no caller of any reaper in the daemon — `grep` shows none).
**Reasoning:**
- The listener task (account.rs:112-121) runs `await_code` with `AUTH_LISTENER_TIMEOUT_S` and then resolves the oneshot with either Ok(code) or Err(Timeout). The oneshot result stays parked in `LoginSession.code_rx` (now ready). Memory is small per session but unbounded over uptime if the CLI is killed mid-flow repeatedly.
- More importantly: if `await_code` succeeded, the bare auth code (still potentially usable until Microsoft expires it ~10 min later) is held in process memory beyond its window.
**Suggested direction:** Stamp `LoginSession` with a `created_at` (Clock) and add a periodic sweeper, or wrap each session in a `tokio::time::timeout` that drops the entry on expiry. `AUTH_LISTENER_TIMEOUT_S + small_grace` is the natural bound. Document the lifetime in the doc-comment.

### F3. 400 token-endpoint errors other than `invalid_grant` are misclassified as `Decode` — BUG
**Location:** `crates/onesync-graph/src/auth/code_exchange.rs:61-72` and `crates/onesync-graph/src/auth/refresh.rs:52-63`.
**Severity:** BUG
**Summary:** OAuth `/token` returns HTTP 400 with `error` ∈ {`invalid_grant`, `invalid_client`, `unauthorized_client`, `invalid_request`, `invalid_scope`, `unsupported_grant_type`, …}. Only `invalid_grant` is mapped to `ReAuthRequired`; every other 400 is converted to `GraphInternalError::Decode { detail: "token endpoint returned 400: {body}" }`, which `map_to_port` (error.rs:125) routes to `GraphError::Decode`. The daemon treats `Decode` as a parse failure rather than a configuration failure → operator sees a misleading error and the scheduler will keep retrying on the next tick.
**Evidence:**
- code_exchange.rs:69-71 — fallback arm formats body into `Decode`.
- refresh.rs:60-62 — identical pattern.
- error.rs:125 — `Decode { detail } => GraphError::Decode { detail }`. No mapping to `ReAuthRequired` or to a "configuration error" variant.
- Microsoft's documented OAuth 2.0 error codes (`https://learn.microsoft.com/azure/active-directory/develop/reference-error-codes`): `invalid_client` (AADSTS7000215 wrong secret/clientid), `unauthorized_client` (consent/policy), `invalid_scope` — none of these is recoverable by simple retry; all three need operator action.
**Reasoning (fault-localization):**
- P1: user mistypes `azure_ad_client_id` in `config.set`, triggers refresh → 400 `invalid_client`.
- P2: daemon should mark the pair `Errored("auth")` / `Errored("permission")` and stop refreshing.
- Actual: daemon emits `Decode { detail: "token endpoint returned 400: {\"error\":\"invalid_client\"...}" }` → `GraphError::Decode` → not retryable but also not flagged as needing re-auth, and not consistent with the spec's table at `04-onedrive-adapter.md:234-236`.
**Suggested direction:** Parse the OAuth error code into a dedicated internal variant (e.g. `GraphInternalError::AuthConfigError { code, description }`) and map it to either `ReAuthRequired` (for `unauthorized_client`, where consent must be re-granted) or to a new `GraphError::AuthConfig` (for `invalid_client`, `invalid_scope`, `invalid_request`) that the scheduler interprets as a permanent pause requiring operator intervention. At minimum, route `invalid_client | invalid_scope | unauthorized_client | invalid_request` to `ReAuthRequired` rather than `Decode`.

### F4. Refresh token rotation: write to vault is unguarded and best-effort — CONCERN
**Location:** `crates/onesync-daemon/src/scheduler.rs:682-685` (and `methods/pair.rs:104-107`, `methods/account.rs:404-407`).
**Severity:** CONCERN
**Summary:** After every successful `refresh::refresh`, the new rotated refresh token is written back with `let _ = self.vault.store_refresh(...).await;` — the result is discarded. If the keychain write fails (locked keychain, sandbox denial, permission prompt cancelled), the daemon continues with a valid access token in memory but the **previous** (now-revoked-on-next-use) refresh token still in the keychain. Subsequent access-token refresh will get 400 `invalid_grant`, demoting the pair to `Errored` and forcing the user to re-login.
**Evidence:**
- scheduler.rs:682-685: `let _ = self.vault.store_refresh(...).await;` — no logging, no error propagation, no audit event.
- `pair.rs:104-107` and `account.rs:404-407` follow the same pattern (`let _ = ...`).
- Microsoft rotates refresh tokens: each /token call returns a fresh `refresh_token` and invalidates the previous one once the new one is used. (Spec confirms this at refresh.rs:7-8.)
**Reasoning:** This converts a recoverable keychain-write transient into an unrecoverable re-auth event. Even if the vault failure rate is low (≪1%), over weeks of operation this is the most likely path to mysterious "had to log in again" reports.
**Suggested direction:** Treat `store_refresh` failure here as a hard error (return the inner error to caller) **or** at minimum log at `warn!` with the account id + error so the operator can see it. Optionally retry the vault write once before failing.

### F5. State-parameter comparison is not constant-time — NIT
**Location:** `crates/onesync-graph/src/auth/listener.rs:69-73`.
**Severity:** NIT
**Summary:** `state != expected_state` is a plain `String` compare. The OAuth `state` parameter is a CSRF guard, not a long-lived secret, but the conventional advice is to use a constant-time compare to avoid leaking timing information. Given:
- The listener is loopback-only (127.0.0.1) and accepts exactly one connection then drops the listener, and
- The state is a one-shot ULID generated per login flow (`new_id::<AccountTag>().to_string()`, account.rs:87) with ~80 bits of randomness,
…the practical exploitability is essentially zero. Calling out for defence-in-depth alignment.
**Suggested direction:** Use `subtle::ConstantTimeEq` (already typically pulled in transitively) or a hand-rolled byte compare. Not load-bearing; only worth doing if/when a security-hardening PR pass happens.

### F6. State value is logged on mismatch — NIT
**Location:** `crates/onesync-graph/src/auth/listener.rs:70-72`.
**Severity:** NIT
**Summary:** On mismatch the error string interpolates both the expected and the received state value with `{state:?}`. The received value is attacker-controlled (anyone who can talk to loopback can craft the query). For a developer chasing a state-mismatch report, logging the **received** value is useful diagnostically — but logging the **expected** state means a subsequent error-message capture by the user (paste-into-issue) could expose the in-flight CSRF token. The in-flight token has minutes-scale lifetime so the window is small; flagging for hygiene only.
**Suggested direction:** Log only "OAuth state mismatch" with no values, or hash-truncate both sides. The full pair can live in `tracing::debug!` if needed.

### F7. Verbatim server-body in `Decode` detail leaks downstream — NIT (defense-in-depth)
**Location:** `crates/onesync-graph/src/auth/code_exchange.rs:70, 82` and `crates/onesync-graph/src/auth/refresh.rs:61, 73`.
**Severity:** NIT
**Summary:** The unparsed response body (which may include `error_description` strings naming the account, claims, or unexpected stack info on misconfigured tenants) flows through `GraphInternalError::Decode { detail }` → `GraphError::Decode { detail }` → daemon logs → potentially user-facing `MethodError` strings (account.rs:215 `format!("token exchange failed: {e}")`). At minimum this means user-visible CLI errors can contain Microsoft-side phrasing the daemon never sanitises.
**Suggested direction:** Keep the body in `tracing::debug!` for diagnostics but redact / truncate before placing it in `MethodError` user-facing strings.

### F8. Redirect URI uses `localhost` but bind is `127.0.0.1` — CONCERN (consistency)
**Location:** `crates/onesync-graph/src/auth/listener.rs:19` binds `127.0.0.1:0`; `crates/onesync-daemon/src/methods/account.rs:83` constructs `http://localhost:{port}/callback`.
**Severity:** CONCERN
**Summary:** Microsoft's identity platform documents that loopback redirects are now matched by **literal string** — i.e. you must register `http://localhost` (and the token endpoint will refuse `http://127.0.0.1` if `http://localhost` is what's registered, and vice versa). The product chooses `localhost` for the redirect URI, which forces the system resolver to map it. On modern macOS this resolves to `::1` (IPv6) first and falls back to `127.0.0.1`. The listener is **only** bound on IPv4 `127.0.0.1` — so if `/etc/hosts` or the resolver returns `::1` first, the browser's GET will hit a connection refusal on `[::1]:port` before retrying `127.0.0.1`. On a stock macOS box with the default `/etc/hosts` (`::1 localhost`), this is reliably the first attempt.
**Evidence:**
- listener.rs:19 `TcpListener::bind("127.0.0.1:0")` — IPv4-only.
- account.rs:83 `format!("http://localhost:{port}/callback")` — relies on resolver.
- macOS default `/etc/hosts` includes `::1 localhost` line first.
**Reasoning:**
- Many browsers (Chrome, Safari) will Happy-Eyeballs both addresses and the IPv4 attempt should succeed quickly, so users may not notice. But the documented Microsoft behaviour (and OAuth 2.0 Native Apps BCP / RFC 8252 §7.3) recommends using `127.0.0.1`/`[::1]` literal addresses, not `localhost`.
- If a future macOS version (or a corporate `/etc/hosts`) drops the IPv4 mapping, the redirect quietly breaks.
**Suggested direction:** Either (a) change account.rs:83 to `format!("http://127.0.0.1:{port}/callback")` and update the Azure-app registration docs to use `127.0.0.1`, or (b) bind dual-stack via `tokio::net::TcpSocket` and bind to `[::]:0` so both `[::1]` and `127.0.0.1` callbacks succeed. Option (a) is the lowest-risk change; (b) matches RFC 8252 dual-stack guidance.

### F9. ID token has no signature, audience, issuer, or nonce verification — CONCERN (by design, but worth flagging)
**Location:** `crates/onesync-graph/src/auth/id_token.rs:1-99`.
**Severity:** CONCERN
**Summary:** `id_token::parse` decodes the JWT payload base64 and reads `tid`, `oid`, `preferred_username`/`upn`, `name` — no signature verification, no `iss` check, no `aud` check, no `nonce` check (no nonce is sent in the authorize URL anyway, account.rs:92-107). The module comment justifies this with *"the token was received over a TLS-authenticated HTTPS connection from the Microsoft identity platform, so we trust the channel"*.
**Reasoning:**
- For the authorization-code+PKCE path, the id_token arrives on the TLS connection from `login.microsoftonline.com`. TLS does guarantee the connection peer, and the id_token is consumed only for **display/identification** (tenant_id, upn, display_name) — *not* for AuthN/AuthZ decisions (the access_token does that).
- However: claim extraction influences `AccountKind::Personal` vs `AccountKind::Business` via `tid == CONSUMER_TENANT_ID` (id_token.rs:73-77). `AccountKind` is used by other parts of the code to route Graph endpoints (e.g. `/me/drive` vs SharePoint), so a manipulated `tid` could in theory steer the daemon to a wrong drive — but only an attacker who can already MITM TLS to `login.microsoftonline.com` could supply such a token, and at that point all bets are off.
- A nonce in the authorize URL + verification on the id_token would catch token-substitution attacks against the loopback redirect step (someone forging the callback). This is a small but real defence-in-depth gap given the loopback listener accepts the very first connection without origin validation.
**Suggested direction:** Add a `nonce` to the authorize URL (random, stored in `LoginSession`) and verify it in `id_token::parse`. Full JWKS-based signature verification is heavier (requires JWKS fetch + caching + key-rotation) and probably not worth it for this use-case; the spec's pragmatic stance is defensible. But the **nonce check is cheap** and forecloses the loopback substitution corner case.

### F10. `url_decode` in listener silently drops invalid UTF-8 byte sequences via `char::from(u8)` — NIT
**Location:** `crates/onesync-graph/src/auth/listener.rs:139-158`.
**Severity:** NIT
**Summary:** `result.push(char::from(byte))` casts each input byte directly to a `char`, treating the bytes as Latin-1 rather than UTF-8. For OAuth `code` / `state` values this is fine — both are URL-safe ASCII per the spec — but a malicious or buggy IdP that returns non-ASCII would silently produce mojibake rather than a decode error. Practical impact is approximately nil; documented for completeness.
**Suggested direction:** Either accept ASCII-only (return Decode error on byte ≥ 0x80) or do real percent-decoding into a `Vec<u8>` then `String::from_utf8`. Not worth changing on its own.

## Cross-cutting observations

- **Concurrency safety of the access-token cache:** `VaultBackedTokenSource.cache: tokio::sync::Mutex<Option<CachedToken>>` (scheduler.rs:542) drops the mutex guard before issuing the HTTP refresh (scheduler.rs:634-641). Two concurrent callers hitting an expired cache will both proceed to call `/token` independently — duplicate refresh requests in flight. Microsoft *rotates* the refresh token on each call, so the second refresh, racing against the first that already rotated, may receive `invalid_grant` and unnecessarily trigger `handle_re_auth_required` (which compounds **F1** badly). The pattern is "check, drop guard, refresh, take guard, write" — a textbook double-refresh. Recommended fix: hold the lock around the whole refresh, or use a `tokio::sync::OnceCell` / single-flight pattern keyed by `account_id`. This is borderline-BUG given F1, but I'm holding it as a cross-cutting concern because reproducing it requires the scheduler to fan out two pair runs against the same account at the same instant — possible but not guaranteed.

- **`unwrap()` / `expect()` in non-test code:** flagged uses in this scope, all already lint-suppressed and documented:
  - `pkce.rs:25` — `getrandom().expect(...)`. Acceptable per project convention (OS CSPRNG failure is unrecoverable).
  - `client.rs:45` — `reqwest::Client::builder()...expect(...)`. Acceptable per project convention.
  - `client.rs:247` — `getrandom().expect(...)`. Acceptable.
  - `login_registry.rs:47, 59` — `Mutex::lock().expect("poisoned")`. Acceptable per project convention.
  - `account.rs:289, 481` — `serde_json::to_value(account).unwrap_or(Value::Null)` is the safe form (no panic). Good.
  - No `panic!`/`todo!`/`unimplemented!`/`unsafe` found in this scope.

- **`request_id: String::new()` in auth errors:** both `code_exchange.rs:66, 77` and `refresh.rs:57, 68` emit empty request-ids for `ReAuthRequired` and `Transient` from the /token endpoint. Microsoft's /token responses do carry an `x-ms-request-id` header — capturing it would meaningfully improve support diagnostics. Not a bug, but a no-cost win.

- **Listener lifetime on success/timeout/cancellation:** `await_code` consumes the `TcpListener` by value (listener.rs:43), so on `Ok`, `Err(Timeout)`, or any panic, the listener is dropped → socket closed. If `account.login.begin` is called but the spawned task is dropped (e.g. tokio runtime shutdown), `listener_sock` is dropped along with the task — port returned to OS. No leaks here. Note however that **port-binding races are not possible** because the OS allocates an ephemeral port (bind on `:0`) — good.

## What looks correct

- **PKCE construction (`pkce.rs`):** Uses `getrandom::getrandom` (OS CSPRNG), 32 bytes → base64url-no-pad verifier (43 chars), SHA-256 → base64url-no-pad challenge. Includes the RFC 7636 Appendix B known-vector round-trip test. RFC-compliant.
- **State entropy:** ULID-based state token (account.rs:87) — ~80 bits of randomness, unique per login, single-use. Adequate as a CSRF guard.
- **Listener bind:** Loopback-only (`127.0.0.1:0`), ephemeral port from OS — no port-collision race. Single-accept-then-drop pattern. Good.
- **Listener timeout:** Wrapped in `tokio::time::timeout(...)` returning `GraphInternalError::Timeout` — clean cancellation drops the listener.
- **Code-exchange happy-path and invalid_grant**: well-tested with wiremock at code_exchange.rs:92-159 and refresh.rs:82-128.
- **`AccountKind` derivation from `tid`:** Consumer tenant constant matches the documented Microsoft value `9188040d-6c67-4c5b-b112-36a304b66dad` (id_token.rs:10).
- **Error mapping (error.rs:110-133):** Per-variant match is total, no `_ =>` wildcard hiding new variants.
- **Documented retry policy on transient/network in `VaultBackedTokenSource`:** Bounded attempts (3), exponential backoff (250ms, 500ms), explicit non-retry of `ReAuthRequired`. Reasonable shape.
- **`re-auth required` audit event:** Emitted with structured `account_id` payload; surfaces to the engine via `pair_upsert` → status change. Operationally observable.

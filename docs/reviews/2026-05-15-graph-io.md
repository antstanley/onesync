### F1. `upload_session` chunk loop ignores 416 `nextExpectedRanges` mid-stream and can rewind to chunk 0 on transient 416 — BUG

**Location:** `/Users/stan/code/onesync/crates/onesync-graph/src/upload/session.rs:131-159` and `199-208`
**Severity:** BUG
**Summary:** On 416, the code re-queries the session, takes `next_expected_ranges[0]`, parses its start offset, and calls `find_chunk_index` to resume — but `find_chunk_index` returns the **next** chunk index after the target, so for `target_offset = 0` it returns `1` (skipping chunk 0). For typical mid-upload offsets it skips the chunk that contains that offset, corrupting the upload.

**Evidence:**

```rust
fn find_chunk_index(chunks: &[bytes::Bytes], target_offset: u64) -> usize {
    let mut off = 0u64;
    for (i, chunk) in chunks.iter().enumerate() {
        if off >= target_offset {
            return i;
        }
        off += chunk.len() as u64;
    }
    chunks.len()
}
```

For `target_offset = 0`, the first iteration has `off = 0 >= 0` and returns `i = 0`. So chunk-0 case is *accidentally* correct. But for `target_offset = 5` with `chunks = [5 bytes, 5 bytes]`: iteration 0 → `off=0 >= 5` false, `off=5`; iteration 1 → `off=5 >= 5` true, returns `i=1`. That happens to be right too. Now `target_offset = 3` with `chunks = [5, 5]`: iteration 0 → `off=0 >= 3` false, `off=5`; iteration 1 → `off=5 >= 3` true, returns `i=1`. **Wrong** — chunk 0 contains bytes 0–4 inclusive, so byte 3 lives in chunk 0; resuming at chunk 1 skips bytes 3–4 entirely and the upload's `Content-Range` arithmetic at line 113 (`offset + chunk_len - 1`) will then claim chunk 1 covers bytes `3-7/total`, which the server rejects or silently corrupts (depending on whether Graph deduplicates by range).

The function also has no way to handle a non-aligned resume offset (a partial chunk): every chunk is assumed to start at the chunk-aligned boundary.

**Reasoning:**

```
PREMISES
P1: 416 from upload-session means the chunk's Content-Range doesn't line up with
    what the server has accepted; the server tells the client which ranges remain
    via `nextExpectedRanges`.
P2: The recovery contract is: "resume from `nextExpectedRanges[0].start`."
P3: Chunks in this code are a fixed slice; `find_chunk_index(N)` must return the
    index of the chunk whose byte-range contains N.

FUNCTION RESOLUTION
- `find_chunk_index` is local to session.rs; no shadowing.
- The 416 branch calls `parse_range_start` then `find_chunk_index`; the parsed
  start is assigned to both `offset` and the index lookup result. `Content-Range`
  on the next PUT is built from `offset + chunk_len - 1`. If the chunk_index
  doesn't align to `offset`, the Content-Range lies about which bytes the body
  contains.

EXECUTION TRACE
- chunks = [5 bytes, 5 bytes], total = 10
- 1st PUT chunk 0 → 416
- GET session: `nextExpectedRanges = ["3-"]` (server already has bytes 0–2)
- parse_range_start("3-") → Some(3); offset = 3
- find_chunk_index(chunks, 3) → walks: i=0, off=0, 0>=3 false, off=5; i=1, off=5,
  5>=3 true, returns 1.
- Loop continues with chunk_index = 1, offset = 3, chunk[1] is 5 bytes.
- Content-Range = "bytes 3-7/10" but the body is the *second* chunk (bytes 5–9).
- Server either returns 4xx (PUT range/body disagree) or stores wrong bytes.

REGRESSION CHECK
- The existing tests only cover the `next_expected_ranges = ["0-"]` case, which
  happens to round-trip because of the off-by-one in the early-return condition.

EDGE CASES
- If `nextExpectedRanges` is empty (server reports completion), the code falls
  through to `return Err(GraphInternalError::InvalidRange { ... })` — but a 416
  with empty `nextExpectedRanges` after the last chunk is the server saying "I
  already have it"; the engine will see InvalidRange and retry forever or fail
  the op.
- The chunk iterator is materialised eagerly into a `Vec<bytes::Bytes>` (line
  106), so it is replayable — that part is fine.

VERDICT: BUGGY for any 416 resume whose `nextExpectedRanges` start does not lie
        exactly on a chunk-boundary, plus the empty-`nextExpectedRanges`
        finish-confirmation case.
CONFIDENCE: high
```

**Suggested direction:** `find_chunk_index` should return the index of the chunk whose half-open range `[cum_off, cum_off + len)` contains `target_offset`, and the loop should re-slice that chunk from `target_offset - cum_off` so the next PUT body matches the Content-Range. Also handle the empty-`nextExpectedRanges` case as "upload complete; ask for the item metadata."

### F2. No `Retry-After` honoring at the adapter; 416 retry has no budget; throttle bucket pause never enforced — BUG

**Location:** `/Users/stan/code/onesync/crates/onesync-graph/src/upload/session.rs:131-159`, `/Users/stan/code/onesync/crates/onesync-graph/src/throttle.rs:36-46`, `/Users/stan/code/onesync/crates/onesync-graph/src/client.rs:107-127`
**Severity:** BUG
**Summary:** Throttling is partially implemented but the `Retry-After` value never causes the bucket to pause. The token bucket refills at a fixed `GRAPH_RPS_PER_ACCOUNT` regardless of whether the server has just told us "back off for 60 s." This means a 429 storm during an upload session (or download / delta) will be retried at the engine level on the next cycle, but in-flight calls on the same `Bucket` will continue burning permits as if nothing happened. The 416 resume in `upload_session` is also unbounded — there is no maximum-retries counter.

**Evidence:**

```rust
// throttle.rs
ticker.tick().await;
if sem_clone.available_permits() < burst {
    sem_clone.add_permits(1);
}
```

The bucket has no `pause_until(deadline)` method; the `Throttled` error is constructed only to surface `retry_after_s` to the engine. Spec `04-onedrive-adapter.md:212-220` says the adapter "Honours `Retry-After` strictly" — but the adapter just reports it. The engine's executor (`engine/executor.rs:39`) treats `Throttled` as a retry signal but does not call back into `Bucket` to pause the bucket, so other parallel calls on the same client continue at full rate immediately.

In `upload_session.rs`, the 416 loop:

```rust
loop {
    ...
    if status == 416 { ... continue; }
    ...
}
```

If Graph returns 416 repeatedly (e.g. a bug in `nextExpectedRanges` parsing, or genuine server confusion), this loop spins forever. There is no max-retries budget local to the session, and no `tokio::time::sleep` between attempts.

**Reasoning:**

```
PREMISES
P1: Microsoft Graph requires that the client back off for Retry-After seconds
    after a 429/503 (spec §Throttling).
P2: The token-bucket in throttle.rs caps long-run RPS at 8 per account, but a
    burst of 16 calls is permitted before the bucket pinches.
P3: When the server says "Retry-After: 60", any of the 16 in-flight calls would
    issue another request immediately under the current design.

FUNCTION RESOLUTION
- Bucket::acquire awaits a single permit; no pause_until or `Throttled` channel.
- check_status returns Throttled to the caller; nothing in the graph crate
  consumes it to pause the Bucket.

EXECUTION TRACE
- T=0: 16 PUT chunks fired against an upload session.
- T=10ms: server returns 429 with Retry-After: 60 on chunk #1.
- T=10ms+ε: chunks #2..#16 are mid-flight on the wire and arrive at the server,
  which now blacklists the connection.
- Adapter surfaces Throttled to engine; engine's executor reschedules the op
  for ~60s later.
- BUT: a separate Pair on the same account, running concurrently, calls
  `Bucket::acquire` and gets a permit immediately. It fires another PUT, gets
  another 429 with a longer Retry-After, escalating Graph's penalty.

REGRESSION CHECK
- Tests use `Bucket::with_rate(100)` to disable the throttle entirely (client.rs
  tests), so this path has no coverage.

VERDICT: BUG. Spec says "Honours Retry-After strictly"; implementation reports
        it but does not pause subsequent calls. Plus the 416 retry has no
        budget — a unit-test for a misbehaving server would loop forever.
CONFIDENCE: high
```

**Suggested direction:** Give `Bucket` a `pause_until(Instant)` (or `pause_for(Duration)`) method; teach `check_status` (or a wrapping retry helper) to call it on `Throttled`. Add a per-op attempt counter to the 416 loop in `upload_session_with_base` (mirroring `RETRY_MAX_ATTEMPTS = 5` from `limits.rs`) and a small sleep between attempts.

### F3. `UploadSession::bytes_uploaded` is dead state — the port-level `upload_session` returns it as `0` and `upload_session_with_base` never reads it — CONCERN

**Location:** `/Users/stan/code/onesync/crates/onesync-graph/src/adapter.rs:233-237`; `/Users/stan/code/onesync/crates/onesync-protocol/src/remote.rs:184-189`; `/Users/stan/code/onesync/crates/onesync-graph/src/upload/session.rs:50-192`
**Severity:** CONCERN
**Summary:** The port returns an `UploadSession { upload_url, bytes_uploaded }`. The adapter sets `bytes_uploaded: 0` unconditionally and `upload_session_with_base` ignores it on entry. Spec `04-onedrive-adapter.md:165-168` promises: "The session URL is persisted in `FileOp.metadata` so a daemon restart can resume without re-uploading." That resume path cannot work today: nothing in the adapter accepts a non-zero starting offset.

**Evidence:**

- `adapter.rs:235`: `bytes_uploaded: 0,` — hard-coded.
- `upload/session.rs:107`: `let mut offset: u64 = 0;` — always starts at 0.

**Reasoning:**

```
PREMISES
P1: After a daemon restart, an upload op may have a half-completed session URL
    persisted with a known bytes-uploaded watermark.
P2: To resume, the adapter would need to query the session for
    nextExpectedRanges, then PUT only the missing chunks.
P3: The current upload_session_with_base assumes the caller starts at offset 0
    with the full chunk iterator.

EXECUTION TRACE
- Persisted state: { upload_url: "https://.../1234", bytes_uploaded: 5 MiB }
- Daemon restart → engine calls adapter.upload_session(parent, name, size)
- Adapter POSTs createUploadSession AGAIN, getting a NEW upload_url, discarding
  the persisted one. Upload starts from byte 0.

REGRESSION CHECK
- No test exercises the "resume after daemon restart" path because the adapter
  trait shape forces re-creation of the session every time.

VERDICT: Behavioural promise in spec is not met by the implementation. Either
        the port should grow `resume_upload_session(upload_url, ...)` or the
        adapter should accept a `bytes_uploaded` seed in `upload_session_with_base`.
CONFIDENCE: high (port-level mismatch)
```

**Suggested direction:** Either drop the spec promise (mark "resume on restart" as future work) or add an adapter entrypoint `resume_upload_session(upload_url, total_size, chunks)` that calls `GET {upload_url}` first to pick up the server's `nextExpectedRanges`, then resumes the loop. Once F1 is fixed, the underlying chunk-index math is already prepared for partial resumes.

### F4. `upload_small`, `mkdir`, `item_by_path`, and `createUploadSession` URLs interpolate `name` / `path` raw — special characters break the URL — BUG

**Location:** `/Users/stan/code/onesync/crates/onesync-graph/src/upload/small.rs:37-40`, `/Users/stan/code/onesync/crates/onesync-graph/src/ops.rs:134-137`, `/Users/stan/code/onesync/crates/onesync-graph/src/adapter.rs:187-192`, `/Users/stan/code/onesync/crates/onesync-graph/src/upload/session.rs:59-62`, `/Users/stan/code/onesync/crates/onesync-graph/src/items.rs:222-234`
**Severity:** BUG
**Summary:** Filenames are interpolated directly into `:/{name}:/content`-style Graph URLs without percent-encoding. Any name with a `#`, `?`, `%`, space, or `:` will either be rejected by Graph or, worse, be misinterpreted (`#` truncates the URL; `?` injects a query string). `items.rs:encode_path` does percent-encoding for `item_by_path` only.

**Evidence:**

```rust
// upload/small.rs:37
let url = format!(
    "{GRAPH_BASE}/drives/{}/items/{parent_item_id}:/{name}:/content?@microsoft.graph.conflictBehavior=replace",
    drive_id.as_str()
);
```

Inputs: `name = "report Q1 2026.docx"` → URL contains a literal space.
Inputs: `name = "v1.0#draft.txt"` → URL becomes `…:/v1.0#draft.txt:/content?…` and reqwest drops `#draft.txt:/content?@…conflictBehavior=replace`. The PUT goes to the wrong path, possibly creating an item named `v1.0`.

Same defect in:

- `ops.rs:134` `mkdir` 409-resolver: `existing_url = "…/items/{parent}:/{name}"`.
- `adapter.rs:188` `createUploadSession`: `format!("…/items/{}:/{}:/createUploadSession", parent, name)`.
- `upload/session.rs:59` (the non-port helper) — same shape.

`items::encode_path` exists but is only called for `item_by_path`. Even there, the spec says paths may include `:` as the *root* separator (`/me/drive/root:{path}`) and the implementation does not address NFC/NFD normalization for files originating from macOS HFS+/APFS (which can deliver NFD).

**Suggested direction:** Centralise an `encode_segment` helper (e.g., `percent_encoding::utf8_percent_encode(name, PATH_SEGMENT)`) and use it everywhere a name flows into a URL segment. For paths, retain `/` as a delimiter but encode `#`, `?`, `%`, `:`, ` `, and non-ASCII. Add normalisation (e.g., `unicode-normalization::nfc`) when accepting names from the local adapter to match Graph's expectations.

### F5. Download is not streamed — buffers entire file in `Vec<u8>` before returning — CONCERN

**Location:** `/Users/stan/code/onesync/crates/onesync-graph/src/download.rs:63-111`, `/Users/stan/code/onesync/crates/onesync-graph/src/adapter.rs:132-139`, `/Users/stan/code/onesync/crates/onesync-protocol/src/remote.rs:174-178`
**Severity:** CONCERN
**Summary:** `RemoteReadStream` is a `bytes::Bytes` wrapper; the adapter accumulates the whole download in a `Vec<u8>` (with redundant `Bytes::from(Vec)` copy at the end) before returning. With `MAX_FILE_SIZE_BYTES = 10 GiB`, this means up to 10 GiB held in memory per concurrent transfer, multiplied by `MAX_CONCURRENT_TRANSFERS = 4` (i.e. up to 40 GiB of RSS). The struct's own docstring acknowledges this: "Wraps a `bytes::Bytes` payload eagerly loaded for the current adapter."

The hash *is* computed during streaming (good), but the comparison happens only after the whole file is in RAM, and the engine then has to write the buffer to disk again. There is no partial-file recovery on disconnect — a 9 GiB download that fails at byte 8 GiB starts over from byte 0 with no state preserved.

Additionally, the streaming code re-allocates `chunk_buf` and copies into `all_bytes` for every block — `chunk_buf.drain(..N).collect::<Vec<_>>()` is an extra full-block copy on the hot path.

**Reasoning:**

```
PREMISES
P1: Spec §Downloads says "streams the body" through `LocalFs::write_atomic`.
P2: The port returns `RemoteReadStream(Bytes)`, an eager buffer.
P3: The engine cannot start writing until the entire body has been collected
    AND its hash verified.

EXECUTION TRACE
- 9 GiB file download.
- Adapter Vec grows to 9 GiB; hash computed concurrently.
- Adapter wraps in Bytes; engine receives RemoteReadStream(Bytes(9 GiB)).
- Engine writes 9 GiB to write_atomic.
- Peak RSS during this single download ≈ 9 GiB.
- If the connection drops at byte 8.9 GiB: returns Network error; engine retries
  from scratch; no resumed download support (Graph storage URLs accept Range,
  but the adapter doesn't issue one).

VERDICT: Behaviour ≠ spec. Memory pressure is real for a 10 GiB MAX_FILE_SIZE,
        especially with 4 concurrent transfers and a Mac mini's typical RAM.
CONFIDENCE: high
```

**Suggested direction:** Redesign `RemoteReadStream` to wrap a `BoxStream<Result<Bytes, GraphError>>` (or hand back a `tokio::io::AsyncRead` directly) so the engine can pipe into `write_atomic` chunk-by-chunk. Move hash verification into a finalisation step the engine calls after the stream is exhausted (or have the adapter return both the stream and a `HashVerifier` handle). Track this as a port-shape change.

### F6. `mkdir`'s 409 fallback `GET` returns a `RemoteItem` that lacks the `folder` facet on at least one Graph response shape — CONCERN

**Location:** `/Users/stan/code/onesync/crates/onesync-graph/src/ops.rs:132-156`
**Severity:** CONCERN
**Summary:** Spec `04-onedrive-adapter.md:202-208` says mkdir uses `conflictBehavior=fail` so the engine can "promote the existing item rather than creating a duplicate." The implementation handles 409 by fetching `…/items/{parent}:/{name}`, but this returns whatever item lives at that name — including a *file* if the user happened to upload a same-named file there. The resulting `RemoteItem` will have `file.is_some() && folder.is_none()`, and the engine's promotion logic will assume it has a folder. There is no type check before returning.

**Suggested direction:** After the 409 `GET`, verify `existing.is_folder()`; if not, surface `GraphError::NameConflict` (the spec's documented path for the "different-kind name conflict" case) rather than handing back a misidentified folder.

### F7. `getrandom` failure is a hidden panic path across six files; the crate-level forbids `panic!` but tolerates `expect("getrandom")` — NIT

**Location:** `client.rs:247`, `adapter.rs:204`, `delta.rs:113`, `download.rs:130`, `items.rs:213`, `ops.rs:281`, `upload/small.rs:77`, `upload/session.rs:213`
**Severity:** NIT
**Summary:** Each module re-implements `new_request_id()` with `getrandom::getrandom(&mut buf).expect("getrandom")`. While `getrandom` failure on macOS is exceptionally rare, the spec's intent (forbid panic paths in non-test code) is technically violated. Eight identical copies of an 8-byte → 16-hex-char helper is also a duplication smell.

**Suggested direction:** Hoist a single `crate::request_id()` helper. Either return `Result<String, GraphInternalError>` (mapping to a `Network`-class transient) or use a fallback like `rand::random()` seeded from a process-start RNG cached in a `OnceLock`. Also, the `client::new_request_id` produces 32 hex chars (16 bytes) while the rest produce 16 hex chars (8 bytes) — inconsistent UUID-ish IDs.

### F8. Delta `extract_token_param` returns the first `token=` match in the *full query string* — fragile when Graph returns `&skiptoken=` first — CONCERN

**Location:** `/Users/stan/code/onesync/crates/onesync-graph/src/delta.rs:100-107`
**Severity:** CONCERN
**Summary:** `extract_token_param` splits the URL at `?` and walks `&`-separated `key=value` pairs looking for `k == "token"`. This works for the canonical `?token=…` deltaLink, but:

- The query value is **not URL-decoded**, so a token containing `%3D` is stored encoded. When passed back to Graph via `delta_page` as `?token={cursor}`, the server may double-decode and treat it as malformed — yielding `resyncRequired`.
- If Graph emits `@odata.deltaLink` with a different parameter name (the older `$deltaToken`, or a future variant), the cursor is silently lost — `delta_token` becomes `None`, and the next cycle does a full re-scan. The spec is explicit (`04-onedrive-adapter.md:139`): "Returns the deltaLink's `token` to the engine."
- The fragment portion (`#…`) is included in the last value if present — no `split_once('#')` guard.

**Reasoning:**

```
PREMISES
P1: Microsoft Graph generates deltaLinks of the form
    `<base>/delta?token=<opaque>`; the opaque can include base64url chars and
    percent-encodings.
P2: Persisting an under-decoded token risks a double-encode round-trip.
P3: Persisting None when no `token=` is found silently turns the next cycle
    into a full re-scan, which spec §Delta queries says must only happen on
    resyncRequired.

VERDICT: CONCERN. The function works on the happy-path test but is brittle to
        Graph URL variations and to percent-encoded tokens.
CONFIDENCE: medium (would need to harvest a sample deltaLink from a real run)
```

**Suggested direction:** Use `url::Url::parse(delta_link).query_pairs()` to get URL-decoded params. Treat absence of a known token parameter as `GraphInternalError::Decode` (an explicit error the engine can log), not as silent `None`.

### F10. Throttle bucket panics if `GRAPH_RPS_PER_ACCOUNT < 1` (division by zero) and silently caps burst at `2 * rate as usize` — NIT

**Location:** `/Users/stan/code/onesync/crates/onesync-graph/src/throttle.rs:30-46`

Also: the burst capacity `(rate_rps * 2) as usize` overflows silently for very high test rates, and the refill ticker fires every `1_000 / rate` ms — for rate 8 that's 125 ms (correct), but for rate 100 (test override) it's 10 ms, which on a busy reactor will be coarser than expected.

**Suggested direction:** Guard against `rate_rps == 0` (debug_assert + saturate to 1). Document that high test rates may be ticker-coarse.

## Cross-cutting observations

- **Port shape is mostly clean.** `RemoteDrive` is a straight 1:1 mapping to Graph endpoints. The one leaky abstraction is `UploadSession { bytes_uploaded }` (F3) and `RemoteReadStream(Bytes)` (F5) — both should be revisited together as a port-shape evolution.
- **Auth boundary is well-respected.** The non-auth adapter consumes `TokenSource::access_token()` and `bearer_auth(token)` everywhere; no `auth/` internals leak into the I/O modules.
- **`unsafe_code` is forbidden crate-wide (`lib.rs:12`).** No violations seen.
- **`expect`/`unwrap` in non-test code:** all are `expect("getrandom")` plus the `expect` on `Semaphore::acquire().await.expect("semaphore should never be closed")`. Each is documented with a justification comment, but they are still panic paths (F7).
- **Test doubles drift.** `FakeRemoteDrive` does not model retry/throttle/429/416 at all. A test asserting that "the engine handles throttling" against the fake is a false positive (the fake never throttles). Consider letting `FakeRemoteDrive` accept an injectable response-script so tests can simulate `Throttled` and `InvalidRange`.
- **`@odata.nextLink` pagination is delegated to the caller (engine).** That's spec-correct, but the adapter has no internal loop helper, so any caller that forgets to follow `next_link` will see partial deltas. The two production call sites (`engine/cycle.rs:90, 139`) appear to do this correctly, but a thin `delta_all_pages(http, token, drive, cursor) -> impl Stream<...>` helper would prevent future drift.
- **Idempotency of POST/PATCH:** `mkdir` (POST) uses `conflictBehavior=fail` + 409 fallback (good). `rename` (PATCH) is server-idempotent by content (good). `subscribe` (POST `/subscriptions`) is **not idempotent** — retrying after a network blip will create a duplicate subscription. The current code lets the engine retry on `Network`/`Transient`; this could double-bill.
- **No request timeout** is configured on the `reqwest::Client` (`adapter.rs:64-67`). A hung TCP connection during an upload chunk will block until the OS-level keepalive fires (minutes). Spec mentions `CYCLE_PHASE_TIMEOUT_MS = 60_000` for cycles, but that doesn't apply to individual reqwest calls.

## What looks correct

- **Error-code surface matches the spec table.** `client::check_status` maps 401/403/404/409/410/412/416/429/503 to distinct `GraphInternalError` variants, and `error::map_to_port` translates each to the right `GraphError` variant (spec §Error mapping). The unit tests in `error.rs` cover every transition.
- **`Retry-After` is parsed correctly** when present, and a 60 s default fills in when absent (`client.rs:280, 332`).
- **Delta tombstones flow through correctly.** The `deleted` facet round-trips into `RemoteItem.deleted` and the engine sees `is_deleted() == true` (test `tombstone_items_have_deleted_flag`).
- **`QuickXorHasher`** matches Microsoft's published rolling-XOR-shift algorithm; the tests confirm determinism and length-XOR finalisation.
- **`GRAPH_SMALL_UPLOAD_MAX_BYTES = 4 MiB` and `SESSION_CHUNK_BYTES = 10 MiB`** match the Graph documented limits (the constant test `assert_eq!(SESSION_CHUNK_BYTES % (320 * KIB), 0)` enforces the 320 KiB multiple).
- **Auth boundary is hermetic.** No `panic!`/`todo!`/`unimplemented!` calls in non-test code beyond the documented `expect("getrandom")`.
- **`forbid(unsafe_code)`** is enforced at the crate root.
- **410 `Gone` short-circuit to `ResyncRequired`** in `delta.rs:76` is in addition to the `check_status` mapping — defensive and correct.
